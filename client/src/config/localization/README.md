# Localization

The localization system provides two things through a single hook: `tr` for translated strings, and `fmt` for locale-aware number and datetime formatting. Both are consumed via `useLocalization()`, which must be called inside `LocalizationProvider`.

```tsx
const { tr, fmt } = useLocalization();
```

---

## Translation ÔÇö `tr`

`tr` resolves a dot-path key against the active locale's translation object and returns the string. The English translation in `en.ts` is the base ÔÇö all other locales must mirror its exact key structure and interpolation variables, enforced at compile time by `ValidateTranslation<Base, Target>`.

**Plain key**

```tsx
tr("common.loading"); // "Loading..."
tr("nav.market"); // "Market"
```

**String interpolation** ÔÇö variables in `{{varName}}` syntax

```tsx
tr("auth.continueWithSelectedWallet", { walletName: "Phantom" });
// "Continue with Phantom"
```

Variable names are extracted from the template string at the type level. Passing the wrong name or wrong type is a compile error.

**Numeric variables** ÔÇö `count`, `min`, `max` are typed as `number`

```tsx
tr("validation.passwordTooShort", { min: 8 });
// "Password must be at least 8 characters"
```

**Pluralization** ÔÇö use `{{singular|plural}}` in the template, then pass `count`

```tsx
// translation: "{{item|items}} selected"
tr("someKey", { count: 1 }); // "1 item selected"  ÔÇö not applicable yet, just the pattern
tr("someKey", { count: 3 }); // "3 items selected"
```

`count` drives plural selection. The pipe-separated pair inside `{{}}` is not treated as a variable name ÔÇö it is a plural selector consumed by `count`.

**React node interpolation** ÔÇö variable names prefixed with `$` accept `ReactNode`

```tsx
// translation: "Don't have an account? {{$createAccount}}"
tr("auth.signUpSuggestion", {
  $createAccount: <a href="/signup">Create a new account</a>,
});
```

When any variable is a React element, `tr` returns `React.ReactNode` instead of `string`. The return type is inferred automatically ÔÇö no cast needed.

**Keys with no variables must omit the second argument entirely.** Passing an empty object to a plain key is a type error.

## Type safety

`TranslationKeyPath` is a union of every valid dot-path in the English translation object, generated by `DotPaths<BaseTranslation>`. Passing a key that does not exist produces an error at the key argument itself, not buried inside a generic parameter.

`FmtStrParams<T>` extracts the set of interpolation variables from a template string literal. The second argument of `tr` is typed exactly as that set ÔÇö missing variables, extra variables, or wrong types all fail at compile time.

`ValidateTranslation<Base, Target>` ensures each locale file structurally matches the base. If a key is missing, renamed, or has different interpolation variables in a locale file, the `defineTranslation(...)` call in that file becomes a type error.

`HasNodeParam<T>` narrows the return type: if any `$`-prefixed variable is present in the template, `tr` returns `React.ReactNode`; otherwise it returns `string`. This means you cannot accidentally pass a node-returning `tr` call where only `string` is accepted.

## Number formatting ÔÇö `fmt.num`

`fmt.num` exposes two notation levels, each with four styles:

```
fmt.num.decimal(value)          standard decimal
fmt.num.currency(value)         standard currency
fmt.num.percent(value)          standard percent
fmt.num.unit(value, "SOL")      standard with unit suffix

fmt.num.compact.decimal(value)  compact decimal  (1.23K, 4.56M ÔÇŽ)
fmt.num.compact.currency(value) compact currency ($1.23K, $4.56M ÔÇŽ)
fmt.num.compact.percent(value)  compact percent
fmt.num.compact.unit(value, u)  compact with unit suffix
```

All functions accept `number | null`. A `null` value returns `"-"`.

The second argument `abs` (default `false`) formats the absolute value when `true`, useful for displaying unsigned changes:

```tsx
fmt.num.percent(change, true); // always positive
```

**Currency and exchange rates** ÔÇö the `currency` style in non-USD locales automatically converts the value using the locale's `getExchangeRate` function before formatting. In `vi`, this multiplies by the USDÔćĺVND rate, so you always pass raw USD values ÔÇö the formatter handles conversion:

```tsx
fmt.num.currency(12.5); // en Ôćĺ "$12.50"
fmt.num.currency(12.5); // vi Ôćĺ "Ôéź312,500"
```

**Unit style** ÔÇö token symbols (`SOL`, `BONK`, etc.) are not valid `Intl.NumberFormat` units. The `unit` style always formats as `decimal` internally and appends the symbol as a plain suffix:

```tsx
fmt.num.unit(1234.5, "SOL"); // "1,234.50 SOL"
```

Fraction digit defaults are defined per locale in `en.ts` / `vi.ts`. For `vi`, `currency` uses zero fraction digits because VND does not use sub-unit denominations.

## Datetime formatting ÔÇö `fmt.datetime`

`fmt.datetime` wraps dayjs. All methods treat the input as UTC and convert to local time before formatting, except `utc` and `iso`. Input can be anything `dayjs` accepts: ISO string, `Date`, Unix timestamp number, etc. A `null` input returns `"-"`.

```
fmt.datetime.date(value)                 locale date pattern, local time
fmt.datetime.time(value)                 locale time pattern, local time
fmt.datetime.datetime(value)             locale datetime pattern, local time
fmt.datetime.utc(value)                  UTC datetime pattern, no conversion
fmt.datetime.iso(value)                  ISO 8601 string
fmt.datetime.relative(value)             relative time ("3 hours ago")
fmt.datetime.fromUnixSeconds(seconds)    Unix seconds Ôćĺ datetime
fmt.datetime.fromUnixMilliseconds(ms)    Unix milliseconds Ôćĺ datetime
```

Each locale defines its own date/time patterns in `defineDateTimeFormat(...)`:

| Locale | `datePattern` | `timePattern` | `dateTimePattern`     |
| ------ | ------------- | ------------- | --------------------- |
| en     | `MM/DD/YYYY`  | `hh:mm A`     | `MMM D, YYYY hh:mm A` |
| vi     | `DD/MM/YYYY`  | `HH:mm`       | `D MMM YYYY HH:mm`    |

Because `fmt` is rebuilt with `useMemo` whenever `lang` changes, all components using `fmt.datetime` re-render automatically with the correct locale patterns on language switch.

## Adding a new locale

Create a new file (e.g. `ja.ts`) that imports `defineTranslation` from `en.ts` and wraps the translation object with it. `defineTranslation` is `defineTranslationWithBase<typeof en.translation>()`, so the returned function validates structural and interpolation-variable parity with the English base at compile time. Define `format` using `defineNumberFormat` and `defineDateTimeFormat` with the appropriate locale code and patterns. Then add the export to `index.ts` under the `locale` object ÔÇö `LangCode` and `TranslationKeyPath` will update automatically.

---

## Adding a new translation key

Add the key to the `translation` object in `en.ts` first. All other locale files will immediately produce a type error at their `defineTranslation(...)` call until the same key is added with matching interpolation variables. There is no runtime key-missing fallback ÔÇö missing keys fail at build time.
