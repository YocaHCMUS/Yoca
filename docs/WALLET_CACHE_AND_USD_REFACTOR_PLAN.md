Wallet data refactor and bug fix plan

This plan targets three concrete outcomes. The first outcome is stopping unnecessary full-period transaction fetches when the requested period is already partially covered in the database. The second outcome is ensuring the service reliably detects and reuses stored records instead of treating them as missing. The third outcome is making token balance history use USD prices from each historical point instead of multiplying all points by a single current price.

## Newly identified root-cause bugs

Four concrete bugs have been identified that explain why repeated fetches persist even after the range-coverage work. They must be fixed before the coverage logic can be considered correct.

The first bug is in walletDataRetriever.ts at the isFullyCovered check. The head-coverage condition compares latestSec from cached rows against requestedRange.toSec, which defaults to the current second. Any wallet whose newest cached transaction is even one second older than now will never satisfy this condition, so every request is treated as having a head gap and the service calls Helius again. The correct fix is to treat the head as covered when the cached head is at least as recent as the head of the requested window, not as recent as the moment the request is being processed. For a requested window whose toSec is the current time, the service should use fetchedAt from the meta table as the head-freshness signal instead of comparing raw transaction timestamps against toSec.

The second bug is also in walletDataRetriever.ts at the tail-coverage condition. earliestSec from the cached rows is used as proof that the tail of the requested window is covered. That assumption fails for sparse wallets and wide windows. If the oldest transaction in the wallet genuinely occurred after requestedRange.fromSec, the cache can be complete and still show earliestSec greater than fromSec, triggering a tail fetch that returns nothing from Helius. The retriever cannot distinguish this case from a real gap unless the covered lower bound is persisted explicitly in the meta table rather than inferred from transaction timestamps.

The third bug is a schema gap in wallet_transactions_meta. The table stores only fetchedAt. With only a freshness timestamp and no persisted covered lower and upper bounds, the service cannot distinguish a 30-day range that was fully synced with a sparse wallet from a case where only a narrower window was ever synced. This is the structural reason that the range-coverage logic in the service layer cannot be made reliable with the present schema. The fix is to add coveredFromSec and coveredToSec columns to wallet_transactions_meta and populate them during every cache write, including writes that persist zero new rows when a head sync finds no new transactions.

The fourth bug is in walletData.service.ts where metadata is only updated when fetchedTransactions.length is greater than zero. When the incremental head check proves there are no new transactions, the service does not record that successful check result. The next request cannot tell that a recent check already happened, so it triggers the same head sync again. Metadata must be updated after every completed sync, including zero-row syncs, with the new coveredFromSec set to the minimum of the existing covered lower bound and the sync lower boundary, and coveredToSec set to the upper boundary of the sync window.

## Implementation phases

The zeroth phase, which unlocks all subsequent coverage fixes, is the schema migration for wallet_transactions_meta. The table must gain two new nullable integer columns named covered_from_sec and covered_to_sec. These represent the earliest and latest second timestamps that have been confirmed as covered by a completed Helius sync for that wallet and chain. They are nullable so that rows inserted before the migration do not break existing reads. Every cacher write path, including zero-row syncs, must upsert these bounds alongside fetchedAt.

The first implementation phase should be a stabilization pass for deterministic cache misses. The chain value written into wallet cache tables must be normalized to the same canonical format used by retriever queries. If the writer uses one chain value and the retriever filters with a different chain value, cache lookups fail and the service re-fetches from Helius even when data exists. This phase should also remove any fixed seven-day assumptions in cache reads when the caller provides a wider period through fromSec. The service should not constrain cache retrieval to seven days when the requested range is thirty days, ninety days, one year, or all time.

The second implementation phase should introduce explicit range-aware cache coverage, now grounded in persisted bounds rather than inferred transaction timestamps. The retrieval contract should use fromSec and toSec. For each wallet and chain, the service should read coveredFromSec and coveredToSec from the meta table and compare them against the requested interval. If the persisted covered range fully contains the requested interval, return from cache only with no Helius call. If the covered range partially overlaps, fetch only the missing edges. If no covered range is recorded, fetch the full requested interval and persist bounds. The comparison for the head edge must use the persisted coveredToSec, not the maximum transaction timestamp from the data rows, and the comparison for the tail edge must use the persisted coveredFromSec, not the minimum transaction timestamp from the data rows. This removes both the false head-gap and false tail-gap conditions described above.

The third implementation phase should add incremental synchronization logic. For head gaps, fetch recent pages until overlap with existing signatures is detected. For tail gaps, fetch backward until the lower boundary is reached. For every fetch path, deduplicate by signature, upsert records, and keep metadata updates consistent with data writes. Metadata must be updated after every fetch path, including the case where zero new rows were returned. The pagination contract should remain aligned with current Helius behavior, including using nextCursor correctly per endpoint semantics.

The fourth implementation phase should fix historical USD valuation for token balance history. The token balance reconstruction in units can remain reverse-accumulation from current balance, but USD valuation must be computed with a timestamp-aware price map. The service should request historical daily prices for the resolved token mint across the requested time window, map each balance point to the nearest valid prior price point, and only fall back to current price when historical price is unavailable for that date. This removes the current behavior where all usdSeries points are derived from one currentPrice value.

The fifth phase should harden fallback and observability behavior. When historical prices are missing for selected dates, the response should still return aligned tokenSeries and usdSeries with deterministic fallback rules. The service should emit structured logs for cache hit ratio, requested range, covered range, fetched page count, fetched transaction count, and number of points priced from historical data versus fallback data. This makes regressions visible and simplifies production debugging.

## Testing

Testing implementation should be expanded from structural tests to behavior-driven service tests. Cache writer tests should validate that coveredFromSec and coveredToSec are persisted on every write including zero-row writes, canonical chain persistence, and signature deduplication. Cache retriever tests should validate that full coverage is determined from persisted meta bounds not from transaction timestamp min/max, that a sparse wallet with no transactions older than fromSec is correctly reported as fully covered when coveredFromSec is at or below fromSec, and that a stale meta row with null bounds falls back to fetch-all behavior. Fetcher tests should validate overlap-stop behavior, boundary-stop behavior, null timestamp skipping, and endpoint-specific cursor usage. Wallet service tests should validate full cache-hit behavior with no Helius call, partial-gap fetch behavior for both head and tail gaps, zero-new-rows sync resulting in updated metadata, and deterministic merge ordering after gap fill.

Token balance history tests should validate that tokenSeries is reconstructed correctly in units and that usdSeries changes when historical prices change over time. A dedicated test fixture should use at least three historical price points with different values so flat valuation bugs are detected immediately. Another test should validate fallback to current price only for missing days while retaining historical prices for available days. Tests should also assert timestamp alignment between tokenSeries and usdSeries for all periods.

Route-level tests should validate output contract and dispatch behavior rather than placeholder assertions. For single wallet plus no token, tests should confirm total-mode response and no token-history service call. For wallet plus token selectors, tests should confirm token-mode response, dual series output per wallet-token pair, and metadata correctness. Multi-wallet and multi-token tests should confirm pair cardinality and series naming consistency.

## Rollout

A one-time data repair step should be included during rollout. Previously stored chain values that do not match canonical query filters must be normalized. Previously stored meta rows that have null coveredFromSec and coveredToSec should be treated as uncovered on first request and will be populated correctly after the first post-migration sync for each wallet.

## Completion criteria

Completion criteria should require that repeated requests for the same wallet and period do not replay full Helius history when coverage already exists, that partial coverage requests only fetch missing boundaries, that a sync returning zero new rows still updates meta bounds so the next request avoids redundant Helius calls, and that token usdSeries reflects historical price variation over time. Completion should also require passing unit tests for cacher, retriever, fetcher, and wallet service, plus route-level tests for balance chart endpoints.
